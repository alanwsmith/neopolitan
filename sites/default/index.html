Welcome To Neopolitan

A plain-text format for websites

I'm designing Neopolitan in the build-in-public style. Most things are solidified but there's still a lot of work in progress. It's far enough along that I'm building this site with it, but I'd hold off doing anything more than playing with it at this point.

Introduction

My site is twenty years old. Pondering those decades got me thinking about the next two. I find myself asking "What do I want to use for the next twenty years?" instead of "What do I want to do for my next redesign?" The answers are very different.

The Format

I've moved my site a half dozen times. Each one involved a painful content migration. I've gone from raw HTML to Perl scripts to Wordpress to PHP with various DB schemas to Markdown to MDX to Org-Mode to a mix of all of the above.

I want aspects of each of those rolled into a single format. I haven't found anything like that. So, I'm rolling my own. It's called Neopolitan and it looks like this:

-> title

Welcome To Neopolitan

-> subtitle

A plain-text format for websites

-> note
>> class: highlighted

The AST (Work In Progress)

The specification for Neopolitin isn't HTML output. It's an AST. The documnets have an explicit way to be parsed but the final output is an implementation detail.

The Engine

I designed Neopolitan to use for my site. The format alone isn't much use without a process to turn it into web pages. I built one that I'm using for this site. It's not so much of a reference as it is a demonstration of what can be done with the format. The source code is in this repo

Content Sections

Neopolitan documents are divided into sections. Each one starts with a -> token followed by the name of the section. The type of section determines what happens with the content inside it. For example, the -> h2 section becomes an <h2> element:

Source
-> h2

Bring your best compass
Output
<h2>Bring your best compass</h2>

While -> list sections use - characters to define list items:

Source
-> list

- Raise the sail

- Shut the hatch
Output
<ul>
  <li>
    <p>Raise the sail</p>
  </li>
  <li>
    <p>Shut the hatch</p>
  </li>
</ul>

HTML Equivalents

Most HTML tags have equivalent sections. Take, -> aside, for example, which works like this:

Source
-> aside

Tear a thin sheet from the yellow pad.
Output
<aside>
  <p>Tear a thin sheet from the yellow pad.</p>
</aside>

Raw HTML

A raw -> html section provides the ability to pass raw HTML directly to the final output

Source
-> html

<form>
    <label for="name">Enter your name: </label>
    <input type="text" name="name" id="name" required>
</form>
Output
<form>
    <label for="name">Enter your name: </label>
    <input type="text" name="name" id="name" required>
</form>

Meta Content

Section with differen types of metadata are included as well. For example, -> attributes sections provide key/value pairs at the page level to the processing engine. -> css and -> script sections push their content into the head of the document to provide increased customization.

Source
-> css

.highlight {
  background: white;
}
Output
<style>
.highlight {
  background: white;
}
</style>

Inline Tags

Several inline tags are provided for styling, linking, etc...

Source
-> p

The <<colt|strong>> reared.
Output
<p>The <strong>colt</strong> reared.</p>

Inline Tag Attributes

Key/value pairs passed to inline tags output as attributes:

Source
-> p

The <<tree top|strike|id: alfa|class: bravo>> waved
Output
<p>The <strike id="alfa" class="bravo">tree top</strong> waved</p>

Inline Shorthand

Shorthand versions of code, em, link, and strong look like this:

Source
-> p

`alfa``

_bravo__

>charlie>https://www.example.com/>

*delta**
Output
<p><code>alfa</code></p>
<p><em>bravo</em></p>
<p><a href="https://www.example.com/">charlie</a></p>
<p><strong>delta</strong></p>

Inline Shorthand Attributes

Attributes can also be added to the shorthand tags:

Source
-> p

`echo`id: foxtrot|class: highlighted`
Output
<p><code id="foxtrot" class="highlighted">echo</code></p>

Inline Code Language Shorthand

Source
-> p

`delta`rust`

`echo`rust|id: sierra`
Output
<p><code class="language-rust">delta</code></p>
<p><code class="language-rust" id="sierra">echo</code></p>

Adding an explicit class along with the language currenly results in two class attributes being added to the output. That is, this would be a problem:

Source
-> p

`echo`rust|class: highlighted`
Output
<p><code class="language-rust" class="highlighted">echo</code></p>

An AST Not An Output

The HTML outputs shown above are from my processing implementation. They're an optional starting point, not a spec. Neopolitan output is an implementation details. The thing that has the spec is the AST that's generated by parsing a document.
