Welcome To Neopolitan

A plain-text format for websites

NOTE: I'm doing the "launch before you're ready - build in public" thing. The ideas are mostly in place but the content and the demo implementation are still in the works.

Introduction

My site is twenty years old. Pondering those decades got me thinking about the next two. I find myself asking "What do I want to use for the next twenty years?" instead of "What do I want to do for my next redesign?" The answers are very different.

The Format

I've moved my site a half dozen times. Each one involved a painful content migration. I've gone from raw HTML to Perl scripts to Wordpress to PHP with various DB schemas to Markdown to MDX to Org-Mode to a mix of all of the above.

I want aspects of each of those rolled into a single format. And, I don't want to have to change it again. I haven't found anything that pulls that off. So, I'm rolling my own. It's called Neopolitan and it looks like this:

-> title

Welcome To Neopolitan

-> subtitle

A plain-text format for websites

-> note

NOTE: I'm doing the "launch before you're ready - 
build in public" thing. Most of the ideas here are
in place but some content is still a work in progress. 

-> h2

got me thinking about the next two. I find myself asking

of "What do I want to do for my next redesign?" The answers 
are very different.

-> attributes
>> date: 2023-04-16 19:06:40
>> type: default 
>> published: y

Dogfooding

I'm not a particular fan of the term "dogfooding". "Drink your own champagne" is worse. Either way, that's what I'm doing. This site is built from the Neopolitan documents and a demo site generator. The sample above is an example. It's what built the first part of this page.

The Specification

I'm still working on the spec. It'll cover all the details of the format. It won't have requirements for output though. Those details should reside in the hands of the implementer. Plus, making standards is a tricky game^xkcd^^.

Some Details

Neopolitan documents are divided into sections. Each one starts with a -> token followed by the section type name. The type determines what happens with the content inside it. For example, the -> h2 section becomes an <h2> element:

Source
-> h2

Bring your best compass
Output
<h2>Bring your best compass</h2>

While -> list sections use - characters to define list items:

Source
-> list

- Raise the sail

- Shut the hatch
Output
<ul>
  <li>
    <p>Raise the sail</p>
  </li>
  <li>
    <p>Shut the hatch</p>
  </li>
</ul>

These examples are based off my demo implementation. The specific HTML is just what I'm using. I think it's a solid choice, but it's in no way a mandate.

Full Section List

There are 46 section types. Some are for content. Others are for metadata. There are sections that put content in the head of the document and others designed to call out to external processes.

Details can be found on the full list.

Section Attributes

One of my main frustrations with other formats is the lack of an ability to add attributes to content. Neopolitan addresses that directly. It's a huge factor behind the design. Each content section accepts attributes defined by lines starting with >> tokens directly after the -> section call. Here's the same -> h2 example from above with an id and a class added:

Source
-> h2
>> id: alfa
>> class: bravo charlie

Bring your best compass
Output
<h2 id="alfa" class="bravo charlie">Bring your best compass</h2>
