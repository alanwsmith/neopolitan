  • h2
    blocks:
    • Introduction
    • The specification for Neopolitan 2.0.0 is test based. Any parser that passes all the test meets the specification.

  • h2
    blocks:
    • Test Types
    • The specification is split into three parts.

  • olist
    blocks:
    • 1. AST Generation
    • 2. Required Sections and Attributes
    • 3. HTML Output

  • h2
    blocks:
    • AST Generation
    • Tests are currently built into the rust code. They include parsing for the shorthands.
    • Sections can have attrs (attributes) and flags. The flags must preserve the order from the source input. Attrs are key/value pairs with no requirement for maintaining a specific order.

  • h2
    blocks:
    • Reserved/Required Sections And Attributes

  • h3
    blocks:
    • Sections
    • TODO: Add details about how documents must start with a section (which means all content is in a section)

  • list
    blocks:
    • - metadata - must not output directly on the page.
    • - neopolitan - Outputs the contents of the section without adding anything beyond the content generated from the body of section. This ensures there's a way to test AST generation from a parser. (Necessary because content can't exist outside a section)

  • h3
    blocks:
    • Section Attributes
    • These attributes must be handled by a parser to meet the spec. They can be used in any section.

  • list
    blocks:
    • - <code> -- created: </code>
    • Value is a date in the RFC 3339 format.
    • - <code> -- updated: </code>
    • Value is a date in the RFC 3339 format.
    • - <code> -- template: </code>
    • This is the way to change the output template for a section. The details of the values and their corresponding outputs is an implementation detail, but the attribute key is reserved.

  • h2
    blocks:
    • HTML Output
    • Neopolitan is designed primarily to generate web content. Output is largely an implementation detail with a required exceptions designed to improve portability between systems.
    • Any generator that passes all the output tests meets the spec. Anything not defined in the tests is left as an implementation detail.
